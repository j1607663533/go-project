# 网页版转盘抽奖系统架构设计文档

## 1. 系统概述
这是一个完整的网页版抽奖大转盘系统，提供用户端参与抽奖和管理端管理活动的完整能力。核心能够实现在后台动态配置奖品概率、上架时间和防刷机制等能力。

## 2. 功能模块设计

### 2.1 用户端 (C端 - 前端)
* **转盘 UI 界面**：采用基于 Canvas 或 CSS3 动画的顺滑转盘效果，展示现有奖品。
* **状态展示**：即时展示当前活动是否开启、用户的“剩余抽奖次数”等信息。
* **防重复点击与防抖**：在点击“抽奖”瞬间即锁定按钮（通过 `isDrawing` 变量设置为 `disabled`），在动画完全结束并弹出结果前，严格禁止用户进行二次点击，防止接口被重复刷单。
* **异步抽奖反馈**：点击“抽奖”后，向后端发起请求计算中奖结果，前端根据结果控制转盘准确旋转到对应的奖品格子并弹出中奖提示。
* **抽奖记录查询**：支持用户查看自己的历史抽奖明细（中奖时间、获得奖品）。

### 2.2 管理端 (B端 - 后台)
* **活动管理**：设置抽奖活动的基本信息，包含：
  * 活动开关
  * 开放开始时间 & 结束时间
  * 用户参与限制（如：每日限抽 X 次，总共限抽 Y 次）
* **奖品池设置**：
  * 奖品名称与图片配置
  * 奖品库存（总数与剩余数）
  * **中奖概率配比**（通过“权重”值来控制该奖品被抽中的几率）
* **记录与统计分析**：
  * 记录所有人的抽奖明细列表（支持按时间、奖品、用户过滤导出）。
  * 简易大盘统计：参与人数、各奖品消耗趋势。

## 3. 数据库表设计 (MySQL / GORM)

在现有的 Go / Gorm 后台项目中，可以拓展出以下三张表：

### 表 1: `lottery_activities` (活动配置表)
| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `id` | int | 主键 |
| `title` | varchar | 活动主题 |
| `start_time` | datetime | 开放开始时间 |
| `end_time` | datetime | 开放结束时间 |
| `daily_limit` | int | 单用户每日限制抽奖次数 |
| `status` | int | 状态（0未启用 1正常激活）|

### 表 2: `lottery_prizes` (奖品管理表)
| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `id` | int | 主键 |
| `activity_id` | int | 所属活动ID |
| `name` | varchar | 奖品名称（如：一等奖、谢谢惠顾） |
| `type` | int | 奖品类型（实物 / 积分 / 谢谢惠顾）|
| `image_url` | varchar | 奖品图片 |
| `total_stock` | int | 初始化库存数量 |
| `left_stock` | int | 剩余库存阶段（防超卖核心） |
| `weight` | int | 中奖权重（配置概率的核心） |
| `sort` | int | 在转盘上的排序位置 |

### 表 3: `lottery_records` (抽奖流水明细)
| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `id` | int | 主键 |
| `user_id` | int | 抽奖用户 ID |
| `activity_id` | int | 参与的活动 ID |
| `prize_id` | int | 抽中的奖品 ID |
| `prize_name` | varchar | 抽中的奖品名称（快照存留） |
| `is_hit` | tinyint(1)| 是否中奖（排除谢谢惠顾） |
| `created_at` | datetime | 抽奖时间 |

## 4. 核心抽奖算法设计 (权重随机)

在服务端响应请求时，不建议直接设置绝对的百分比（例如 10%），而是应该**基于剩余库存和权重**计算，确保配置能灵活调整且控制库存。

**后端抽奖执行流程：**
1. **活动有效性校验**：当前时间是否在开放时间内？判断 `start_time` ~ `end_time`。
2. **频控校验**：查询 `lottery_records` 表，查看当前用户当日抽奖次数是否超过了 `daily_limit`。
3. **查出有效奖品**：拉取当前活动的所有配置奖品，**过滤掉库存（`left_stock <= 0`）已经为空的奖品**。
4. **按权重抽奖 (`Weighted Random`)**：
   * 将所有有效奖品的 `weight` （权重）相加，得出 `totalWeight`。
   * 生成一个 `0` 到 `totalWeight - 1` 的随机整数 `R`。
   * 遍历奖品列表，累加它们的权重值 `sum`，当 `R < sum` 时，即可确定落入该奖品区间。
5. **扣除库存与落库操作**：
   * 开启数据库事务（或 Redis Lua）。
   * `UPDATE lottery_prizes SET left_stock = left_stock - 1 WHERE id = ? AND left_stock > 0`（利用乐观锁防止超卖）。
   * 如果更新成功，则向 `lottery_records` 插入一条记录；否则回退事务重试抽奖，或降级为“谢谢惠顾”。

## 5. API 接口设计规划

### 5.1 C 端前端用接口
* `GET /api/v1/lottery/info` 
  * 作用：获取当前转盘面板数据。
  * 返回：开放时间是否正常、剩余可用抽奖次数，以及转盘展示需要的 `奖品列表（id, 名称, 排序, 图片）`。
* `POST /api/v1/lottery/draw`
  * 作用：用户点击“开始”的动作。响应中奖结果。
  * 返回：中奖的 `prize_id`，以便前端驱动转盘停止到对应区域。
* `GET /api/v1/lottery/records/my`
  * 作用：我的中奖列表详情。

### 5.2 B 端后台管理接口
* `GET /api/v1/admin/lotteries`、`POST /api/v1/admin/lotteries`：维护活动主表。
* `GET /api/v1/admin/prizes`、`PUT /api/v1/admin/prizes`：配置具体奖品信息、补库、设置高低概率。
* `GET /api/v1/admin/lottery-records`：查阅大盘统计所有人的抽奖行为。

## 6. 技术重点保障 (高频注意事项)
1. **防止超发 (超卖)**：在多并发情况下，一定要确保不能把奖品发超。基于当前的 `MySQL + GORM`，必须在抽中时使用行锁或乐观锁（`UPDATE xx SET count=count-1 WHERE count > 0`）。
2. **保底机制（谢谢惠顾）**：将“谢谢惠顾”的奖品做成**无限库存**，一旦实物奖品被抽光，或在高并发拿不到行锁时，自动兜底发放“谢谢惠顾”。
3. **前端防流氓**：前端在请求期间按钮 `disabled` 且做防抖处理。后端校验不可少。
